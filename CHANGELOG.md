The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/). ### Fixed
- **еҺҶеҸІйҖҡзҹҘ detail no longer shows the body under еҸ‘еёғж—¶й—ҙ** (the ж Үйўҳ/еҸ‘еёғж—¶й—ҙ "swap" the captain saw on entry-mac). A course notice opened from `AnnouncementsHistoryDialog` rendered the entire announcement body as its еҸ‘еёғж—¶й—ҙ, with ж Үйўҳ a 20-char snippet of that same body. Root cause is in pypku3b's `parse_announcements` (vendored, `blackboard.py`): for an announcement with **no `<h3>` title**, the no-h3 *fallback* branch derives `title` from the first 20 chars of the body and sets `time` to the first `<p>`'s text вҖ” which for these notices *is the body*, so `posted_time` becomes the whole body. Our tool's `_posted_at` then surfaced it verbatim (it only stripped a `еҸ‘еёғж—¶й—ҙ:` prefix, absent here). Fixed in the tool layer (`src/tools/pku3b_announcements.py`) since `vendor/` is a subtree and off-limits: `_posted_at` now returns a value only when it carries the `еҸ‘еёғж—¶й—ҙ` marker (which genuine times always do вҖ” the h3 branch only lifts a time from a `еҸ‘еёғж—¶й—ҙ`-bearing sibling) or is a short bare date line; a marker-less free-text blob (the mis-parsed body) is dropped to `None`. The UI already omits еҸ‘еёғж—¶й—ҙ when `posted_at` is falsy, and `posted_date` was already correctly `None` for these rows, so the symptom is fully cleared. No wire/schema change, no data reset needed (detail recomputes `posted_at` live from `posted_time`). Tests: `test_pku3b_tools.py` (`_posted_at` unit cases + body-blob dropped in list & detail). (worktree `fix-announcement-posted-time-leak`)
- **еҺҶеҸІйҖҡзҹҘ detail resolves вҖ” announcement ids are now content-stable** (supersedes the earlier all-term-retry fix, which did not hold up in real testing). Clicking a course notice in the `AnnouncementsHistoryDialog` showed `announcement with id вҖҰ not found`. Root cause: the id the tool exposed was pypku3b's **positional** value вҖ” `content_hash(course_id, "{course_id}_{idx}")`, where `idx` is the notice's index within one scrape вҖ” so any new/deleted/reordered notice shifts every later id, and the id persisted in `data/announcement_history.json` then matches nothing on a fresh re-list; no re-list or all-term crawl can recover a positional id (which is why retrying couldn't fix it). Fixed in the tool layer (`src/tools/pku3b_announcements.py`): `_stable_id` derives the id from content вҖ” `blake2b` over `(course_id, normalized_title)`, deliberately **excluding `posted_date`** вҖ” used as the canonical `id` in `_to_record` and matched in `_show`, so a re-listed notice re-derives the same id. `_show` dual-matches `{stable_id, positional_id}` so legacy history rows that haven't drifted still resolve, and the all-term retry is kept as a wider candidate set. **Note:** an interim fix hashed `(course_id, title, posted_date)` (falling back to `(course_id, title)` when undated) and *still* failed in real use вҖ” pypku3b drops the еҸ‘еёғж—¶й—ҙ for ~half of announcements and the per-course scrape is TTL-cached ~1h, so store-time and show-time are different scrapes; the `if date else` branch turned a datedвҶ”undated flip into a total hash mismatch (`announcement with id 7b1f39f01a937eb3 not found`). Dropping the date entirely is the durable fix вҖ” title is the only stable content identifier available. The schema / return shape / `ToolResult` wire format are unchanged (only the *value* of `id` changed), and the GUI needs no edit (it passes `item["id"]` blindly). One-time on upgrade: delete `data/announcement_history.json` so rows re-accumulate with the date-free id (old rows won't match). Residual (not tool-fixable): two announcements with the same title in one course now collide (rare; the intermittent date didn't reliably distinguish them either). Tests: `test_pku3b_tools.py` (content-determinism, **date-flip-still-resolves** вҖ” the regression the interim fix lacked, legacy dual-match, genuine-miss still errors). (worktree `pku3b-announcement-detail`)

Fixed upstream in `RizzoHou/plib-cli` (add `agree_terms=y` to the POST) and pulled into `vendor/plib-cli` via `git subtree pull --squash` вҖ” never edited under `vendor/`. Verified live on entry-mac through the real tool: `quota`вҶ’`{download_remaining: 10}`, `search й«ҳзӯүж•°еӯҰ`вҶ’10 results. Tests: `test_plib*` (12) + plib-cli's own suite (24) pass.
