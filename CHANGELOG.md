# Changelog

All notable changes to PKU Captain are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/). The project is pre-release (no Semantic Versioning yet), so entries are grouped by the date a change landed on `main`, newest first. Deep design rationale and code invariants live in `CLAUDE.md`; this file records *what changed*.

## [Unreleased]

### Added
- **Model switcher (DeepSeek V4 Pro ⇄ Kimi K2.6) + first-class vision doc_read** (**BREAKING: integration contract**): reworks how vision works so the chat brain reads documents *itself* instead of a "Kimi-converts-PDF-to-text" middleman. **(1) Kimi K2.6 chat brain**: `KimiProvider` is now a full chat brain mirroring `DeepSeekProvider` — real SSE `stream_chat`, thinking mode (`{"thinking":{"type":"enabled"}}`, returns `reasoning_content`), tool calls, usage parsing, `context_window = 256_000`, default model `kimi-k2.6` on `https://api.moonshot.cn/v1` (all verified live). **(2) Switcher**: a `QComboBox` in the chat header (shown when ≥2 brains have keys); `src.core` gains `DEFAULT_CHAT_MODEL` / `available_chat_models` / `build_chat_llm` / `apply_chat_model`. Switching swaps `agent.llm` and **opens a fresh chat (reset-on-switch)** so no conversation ever mixes models — which dissolves both the image-choke and the cross-model `reasoning_content` replay hazard. The context meter auto-adapts (1M ⇄ 256k) via `agent.llm.context_window`. **(3) Image-injection doc_read**: `doc_read` no longer calls a vision model; it renders the pages and returns them via the new `ToolResult.images`, and `Agent.turn()` injects them as a multimodal `user` message (`Conversation.add_user_parts`) **after** all tool results so the Kimi brain reads the actual pages. It registers **only while the active brain is Kimi** (text brains can't read images), toggled by `apply_chat_model`. **(4) Auto-switch**: `VisionRouter` (keyword heuristic) routes a 培养方案/文档 question asked on DeepSeek into a fresh Kimi chat. **(5) Dashboard**: the 文档库 dialog's 让 Captain 阅读 keeps a self-contained text answer via a new encapsulated `DocBaseReader` service (render → ask Kimi → text), injected like `memory_learner`; `build_vision_llm()` now uses `kimi-k2.6` and only feeds this reader. **Also fixed a latent SSE bug** in both providers: `iter_lines(decode_unicode=True)` corrupts multi-byte UTF-8 split across a chunk boundary (Kimi's long Chinese `reasoning_content` triggers it every time) — switched to raw-byte line iteration + explicit UTF-8 decode. Verified live end-to-end: on Kimi, `doc_search`→`doc_read`(6 pages)→injected images→Kimi answered 138 总学分 with a correct credit breakdown; context went 3.6k→22.8k of 256k. Tests: `test_kimi_provider.py`, `test_agent_vision_inject.py`, `test_vision_router.py`, `test_chat_panel_model_switch.py`, updated `test_doc_base.py` / `test_bootstrap_docbase.py` / `test_session_store.py` / `test_dashboard_dialogs.py` (362 pass). (worktree `doc-base`)
- **Doc base — find + read tools and browseable UI** (replaces the embedding knowledge base end-to-end; **BREAKING: integration contract**): the second half of the doc-base switch. Two new `Tool` subclasses in `src/tools/doc_base.py`: `doc_search` does a lexical ranked search over `doc_base/manifest.json` (title ≫ breadcrumb ≫ volume, with CJK-bigram overlap so an unsegmented query like `数学科学学院` still ranks; empty query → browse the whole corpus) — it reads only the committed manifest, so it registers in **every** mode including offline; `doc_read` renders a document's PDF pages to PNG via `pdftoppm`, base64s them, and has a **vision LLM (Kimi, `moonshot-v1-32k-vision-preview`)** read the tables/flowcharts and answer an optional `question`, returning only the distilled text — so the dense PDF never enters the main agent context. `doc_read` registers **online and only when a Kimi key is present** (the first agent consumer of `KimiProvider`); it caps at 24 pages and takes a `pages` range (e.g. `1-6`) to slice longer documents. Bootstrap gains `build_vision_llm()` and a doc-base line in the system prompt (培养方案/选课/辅修 questions → `doc_search` then `doc_read`, don't guess credit numbers). GUI: the 知识库 header button + `KnowledgeSearchDialog` are replaced by a 文档库 button + browseable `DocBaseDialog` — a `QTreeWidget` of 卷→学部→院系→专业 with a filter box (re-runs `doc_search`), double-click / 打开 PDF opens the real PDF in the OS viewer, and 让 Captain 阅读 runs `doc_read` and shows the answer in `DocReadResultDialog` (gated on `doc_read` registry membership). `build_agent` / `MainWindow` **drop the `enable_knowledge` param** and the `--rag` flag is removed (the doc base has no index to build, so no opt-in); the embedding `KnowledgeSearchTool` is no longer registered, but the `src/rag` infra (`KnowledgeBase` / `Embedder` / `Source`) is kept as an OOP-pillar artifact. Verified live end-to-end: DeepSeek autonomously chained `doc_search`→`doc_read` and answered a 培养方案 学分 question correctly (138 总学分 / 24 核心课学分) from the Kimi-read table. Tests: `tests/test_doc_base.py` (search ranking / browse / volume filter / page-spec parse / read multimodal message + truncation, renderer mocked) + `tests/test_bootstrap_docbase.py` (registration gating). (worktree `doc-base`)
- **Doc base — split corpus** (`scripts/split_doc_base.py` + `doc_base/`): the first half of replacing the embedding knowledge base with a "split PDFs + vision read" doc base (curriculum PDFs carry charts/tables an embedder can't parse). The script splits the four 教务部 source PDFs in `doc_base/original/` into 282 small, topic-scoped PDFs under readable `学部/院系/专业.pdf` hierarchies, plus `doc_base/manifest.json` (every output's source, breadcrumb, title, page range, kind). Per-volume strategy (top of script, `SOURCES`): 培养方案 文/理科卷 split at every PDF outline node via `qpdf --json` (leaves→pdf, 院系 internal nodes→dir + `_概述.pdf` intro, pure 学部 title dividers dropped+logged); 选课手册 and 辅修/双专业 卷 have unreliable/absent bookmarks → hand-verified `EXPLICIT_SECTIONS` tables (选课手册 from text-scanned heading pages — two qpdf dests drifted a page; 辅修/双专业 from its printed 目录, PDF page = printed page + 12 offset, split at 院系 level since the 102 plans flow mid-page and can't be per-major page-split). Extraction is lossless `qpdf --pages`; each output is then re-subset by a Ghostscript pass (`/prepress`, page-count-guarded) cutting size ~4× (qpdf copies the document-wide CJK font subset into every split). The script self-checks per volume: `sum(emitted pages) + dropped == source pages`, and no output-path collisions. Boundaries were spot-verified by rendering first/last pages (text, charts, tables intact). New build deps: `qpdf` + `poppler` + `ghostscript` (system CLIs, documented in `docs/setup_zh.md`). The doc-base **tools/UI** (find + read PDF, replacing the KB search tool and its dashboard entry) are the next step, not in this change. (worktree `doc-base`)

### Fixed
- The **first** thinking window of a session no longer renders too tall (half-empty) when thinking visibility is on; its height now auto-fits like later windows. Root cause: Qt applies the body's QSS font (`#InlineThinkingBody { font-size: 11px }`) only at polish time, which for the first window lands *after* the streaming burst that sized it — so it was measured against the larger default font and kept a stale height no later delta corrected. Fix: `InlineThinking._sync_size_to_text` now calls `self._body.ensurePolished()` before measuring, forcing the stylesheet font to resolve synchronously (a cheap flag-check once polished, so streaming stays O(delta)). Regression test reproduces it under the real app stylesheet (`tests/test_chat_panel_thinking.py::test_thinking_height_uses_polished_stylesheet_font`). (worktree `thinking-window-fixing`)
- The 教务更新 card empty state no longer shows the redundant duplicate "暂无教务部新内容" line — only the single summary line remains. (worktree `dean-notice-adding-and-caching`)
- DEAN items no longer vanish on the next (empty) poll: the new `DeanInboxStore` accumulator resolves the `DeanUpdatesCard` "replaces-not-accumulates" v1 limitation (PR #5). Downloads / openinfo rows are now openable too — `_normalize_item` reads their `download_url` field (they expose the link there, not `url`). (worktree `dean-notice-adding-and-caching`)
- Multi-session course cells (the same course concatenated at different weeks/rooms, e.g. 程序设计实习) no longer leak the next session's text into the previous one's 考试信息. `_clean_course_info` splits on the `<title>(主)` marker via `_split_sessions`, bounds each field to its session, and merges all sessions' 上课信息. (worktree `fix-course-note-absence`)
- `InlineThinking` sliding window now auto-sizes its **height** to the reasoning length (≤160px cap), matching the existing width auto-sizing — short thinking no longer renders in a tall half-empty box. Height is counted from font metrics (`_sync_size_to_text`) since QPlainTextEdit's `documentSize` reads stale right after a width change; a `_saturated` flag keeps streaming O(delta) once both caps are hit. (worktree `fix-thinking-visibility-problems`)
- 讲座推荐 dashboard card now shows only today-or-future lectures, sorted earliest-first, and can reveal them all. Root causes: the fetch was capped at `limit: 5` (so 展开全部 could never reveal the rest), and the card rendered the raw earliest-first list with no upcoming filter (so it showed past lectures).
