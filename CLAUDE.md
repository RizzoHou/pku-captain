Scaffolded but not yet running end-to-end. `src/` is the Python package (importable as `src`; modules use intra-package relative imports so the name `src` rarely shows up in code) and houses the four OOP hierarchies in subpackages — `tools/`, `workflows/`, `llm/`, `rag/`, plus `core/` (agent kernel — `Agent` + `Conversation`) and `ui/` (PyQt6 desktop window). `pyproject.toml` declares the build (hatchling, `packages = ["src"]`) + runtime deps (PyQt6, requests, numpy — embeddings go through a hosted API, so there is no torch/transformers stack) + dev deps (pytest, ruff, mypy). The deeper end-to-end probe is `scripts/smoke_deepseek.py` — it drives a real DeepSeek round-trip with ClockTool through the agent loop and is mandatory after any change to `core/`, `llm/`, or the tool-call wire format. `src/cli.py` (`python -m src.cli`, flags `--offline` / `--show-reasoning`) is the matching interactive REPL on the same loop — it remains the contract-conformance probe for the GUI seam because it goes through `build_agent()`. One reference offline subclass per hierarchy exists (`ClockTool`, `EchoLLMProvider`, `StaticSource`, `HelloWorkflow`) so the loop and tests can run without API keys or live PKU endpoints. The headliner LLM (`DeepSeekProvider`) is already in — captain absorbed it because the model config + thinking-mode wire format have project-wide implications. The Week-1 Tool work shipped together for the same reason: `src/tools/pku3b.py` (shared subprocess wrapper — locates the binary, runs with timeout, ANSI-strips stderr; **not** itself a Tool subclass) and `PKU3bAssignmentsTool` (calls `pku3b assignment list --format json` and `json.loads` the structured output directly — no regex, no rendered-text parsing; surfaces course id, assignment id, raw + ISO-8601 deadlines, completion flag, descriptions, and attachments with their Blackboard URIs).

The RAG retrieval chain (`src/rag/embedder.py` — `Embedder` ABC + `APIEmbedder`, which embeds via Alibaba DashScope `text-embedding-v4` over its OpenAI-compatible endpoint using `requests` (no `openai` SDK, no local model); `src/rag/knowledge_base.py` `KnowledgeBase` — SQLite chunk store + float32 BLOB vectors + numpy cosine search, and `KnowledgeSearchTool`) — note the `Embedder` hierarchy intentionally ships **only** the online `APIEmbedder` (no offline reference subclass, unlike the four main hierarchies) because RAG is online-only and opt-in; tests use an in-file fake embedder; two `Source` subclasses (`DeanSource`, `CalendarSource`) wired through `build_source_registry()`; `MemoryTool` (persistent preference store under `data/`; the `MemoryStore` backend lives in `src/core/memory.py`.

**Memory is folded into responses, not only tool-reachable**: bootstrap shares **one** `MemoryStore` instance between `MemoryTool(store=...)` and `Agent(memory=...)`, and each `Agent.turn()` iteration merges the current entries — via `render_memory_context()` — into the *leading* system message of a snapshot copy. It is never appended as a second system turn (DeepSeek expects one) and never written into `Conversation`, so the block refreshes after a mid-turn `set` yet rendered history stays clean. Two invariants when touching this: keep the store **shared** — a second `MemoryStore` on the same path silently no-ops because it loads once at construction and never re-reads — and keep the block out of `Conversation`.

**Natural-language input becomes memory without a manual key**: `MemoryStore.remember(text)` stores free text under an auto-derived content-hash handle (`note-<sha1>`) — the keyless producer of the *same* entry kind as `set` (no `note`/preference split; `render_memory_context` stays uniform), idempotent on identical text, no semantic overwrite. `MemoryTool` exposes it as the `remember` action; the system prompt tells the agent to prefer `remember` for facts learned mid-conversation (name, location, language, schedule) and reserve `set` for replacing a value under a known key. The dashboard `MemoryDialog` accordingly dropped its key+value form for a single natural-text box → 记住 button (signature stays `(tool, parent)` plus an additive `learner=` — `test_dashboard_dialogs.py` constructs it positionally).

**The dashboard box also does smart LLM extraction**: `MemoryLearnService(llm, store)` (`src/core/memory_learn.py`, exported from `src.core`) is a dedicated *stateless* extraction — prompts the LLM for a JSON array of atomic facts and stores each via `remember` — **not** a detour through `Agent.turn()` (no chat-history pollution / event noise, same result). `MainWindow` builds it from the agent's **own** `llm` + `memory` (shared store, so dashboard- and chat-learned facts coincide) and injects it into `DashboardPanel(memory_learner=...)` → `MemoryDialog(learner=...)`; with a learner the 记住 button runs the split through `run_async` (LLM call) and shows "已记住 N 条", without one (offline) it stores verbatim inline. The JSON parser (`_parse_facts`) **is the feature** — it must tolerate ```code fences``` and prose, or every real reply silently degrades to one verbatim blob; offline `EchoLLMProvider` never yields JSON so it falls back to verbatim with no special-casing. Tradeoff: free-text notes don't overwrite (`我不喝咖啡` won't replace `我喜欢咖啡`; delete the stale one from the list). Backend tests: `test_memory_store.py` / `test_memory_tool.py` / `test_memory_agent.py` — the last uses a scripted LLM to prove a learned fact folds forward into the next call, and asserts `build_agent` shares one store), `LectureTool` (curated snapshot at `src/tools/data/lectures.json`, live-source-shaped schema), `PKU3bAnnouncementsTool`; and `KimiProvider` (vision LLM channel).

The dashboard 树洞消息 card **stays in sync with that auto-checking**. Two parts.

(1) **Accumulate, don't replace**: `monitor.check()` returns only the delta since the dashboard's **own** cursor (`secrets/treehole/state.json` — deliberately **separate** from the notifier's `data/treehole-notify-state.json`; concurrent writes are benign), so replacing per-poll made a reply **vanish on the next empty poll**. Instead a persisted `TreeholeInboxStore` + pure `merge_treehole_updates` (both in `treehole_updates.py`) union by pid/cid into `data/treehole_inbox.json` and keep unread visible until the user opens `TreeholeMessagesDialog` (mark-as-read → `clear`); the inbox is **in-memory by default** (`path=None`, so tests + offline never touch `data/`), injected into `DashboardPanel(...)` by `MainWindow`. `set_treehole_updates` / `set_error` merge-and-render-from-inbox rather than overwrite, so a transient poll error keeps the accumulated rows + badge. Sibling **`TreeholeHistoryStore`** (`data/treehole_history.json`, append-only, per-comment by (pid,cid)) takes the **same raw poll** but is **never cleared on read** — backs the dialog's 历史消息 tab (newest-first, + 清空历史); feed from the poll, not the post-clear inbox (count-only polls / per-poll `limit` are gaps).

(2) **Auto-refresh on the notifier's cadence**: a `MainWindow` `QTimer` re-polls the card while notifications are enabled+supported+**logged-in** (`_reconfigure_treehole_sync`, re-read on startup + on `treehole_settings_changed` when the dialog closes), via a **quiet**, busy-guarded `run_async` poll (no flicker/spam). Limits: the card re-derives via its *own* poll, not the daemon's Notification-Center list (count-based trigger can drift); app-closed replies still surface on reopen because the cursor is frozen while shut. **Don't** unify the (now three) state files or revert accumulation to replace (re-introduces the two-poller race / vanish-on-poll bugs). Tests: `test_treehole_inbox.py` / `test_treehole_history.py` (stores) + `test_treehole_sync.py` (accumulate / mark-read / history-survives-clear / error-preserve, timer gating).

The GUI is a **2-panel** layout (`dashboard | chat`, tool calls inline in the chat flow via `InlineToolCall`; `ToolTracePanel` is retained-but-unmounted) and added dashboard dialogs for treehole/P-Lib/announcements/lectures/reminders/memory/knowledge.

**Dashboard dialogs follow integration-contract §1+§5**: they obtain tools from the injected `ToolRegistry` (`tools.find(name)`, `name in tools`) — never by constructing `Tool` subclasses — with online-only entries gated on registry membership (so offline never touches a network/subprocess tool), and every blocking dialog call goes through `src/ui/tool_call_worker.py`'s `run_async()` onto a `QThreadPool` to keep the window responsive.

**Dashboard refresh is scoped, not global**: the header 刷新 button fires the panel's `refresh_requested` (reloads all cards), but each card's own 刷新 button (and a post-login `auth_changed`) must emit `partial_refresh_requested([tool_key])` so `DashboardWorker.refresh(keys=...)` reloads only that card — wiring a card's refresh to the global `refresh_requested` reintroduces the "one click refreshes everything" bug.

**Dashboard refresh is also parallel, not serial**: `DashboardWorker.refresh` fans the selected tools out over a `ThreadPoolExecutor` and emits each card's result via `as_completed` from the worker thread, so startup time is the slowest single card, not the sum. Concurrent refresh is safe by **error isolation**: `_invoke` swallows every exception and each card reports independently via `item_error`, so one card's failure never crashes the dashboard or corrupts a sibling. The one shared resource within a single refresh is pku3b's content-addressed cache: on a cold/stale session `pku3b_assignments` + `pku3b_announcements` both target the `Blackboard::_get_courses` cache key and may each fetch + re-login, worst case a refetch; `cfg.toml` is read-only on these paths, so no credential race. The Python-side creds injection is concurrency-safe regardless — `PLibMaterialsTool` builds env per call via `os.environ.copy()` + `env=`, never mutating the global. `tests/test_dashboard_worker_refresh.py` guards this with a `threading.Barrier` test that a serial loop cannot pass.

**Dashboard cards are cached to disk for instant reload**: `DashboardCache` (`src/core/dashboard_cache.py`, built via `build_dashboard_cache()`, lazily re-exported) saves each card's **raw tool payload** to `data/dashboard_cache/<tool_key>.json` — one dumb, lock-guarded file per card, mirroring `SessionStore` (no in-memory mirror; the GUI owns change-detection). `MainWindow` paints from `load_all()` at startup (`_seed_dashboard_from_cache`), then fires a **silent** refresh (`_start_refresh([], silent=True)`) that suppresses the spinner on already-cached cards and repaints only the ones whose data changed — a canonical-JSON `_signature` compare, with `_should_render` enforcing the invariant that a card put into the loading state (tracked in `_loading_keys`) always renders so it can never strand on the spinner. Explicit header/per-card refresh stays non-silent (always spins + repaints, for click feedback). A refresh error keeps the last-good cached card visible with a status-bar note rather than wiping it to an error — so an offline launch shows the prior online cache (deliberate; the `离线模式` subtitle signals it). Caching **raw** payloads (not rendered widgets) means time-relative renders like `upcoming_assignments` / `upcoming_lectures` (讲座推荐 = today-or-future only) re-filter against today's clock on repaint. `tests/test_dashboard_cache.py` covers the store plus the pure `_signature` / `_should_render` decision (full MainWindow wiring validated headless, not committed).

**Chat renders one bubble per assistant segment, not per turn**: a tool-using turn runs multiple LLM iterations, each emitting an assistant text segment then tool calls, so `MainWindow._on_agent_event` calls `ChatPanel.finalize_assistant_segment()` on every `tool_call` to lock the current streaming bubble in place before the tool row — without it, later segments overwrite the first bubble (the `final` event's `add_assistant_message` rewrites it) and the last reply renders above its tool calls instead of below; `tests/test_chat_panel_segments.py` drives real `AgentEvent`s through `_on_agent_event` (offscreen Qt) to guard the ordering.

**Chain-of-thought ("thinking") is optionally visible, off by default**: `DeepSeekProvider.stream_chat` yields `ChatStreamEvent(reasoning_delta=...)` per reasoning token (plus accumulating `reasoning_content` for replay), `Agent.turn()` forwards these as an additive `reasoning_delta` AgentEvent (contract §3; before `assistant_delta` per segment), dispatched by `MainWindow._on_agent_event` to `ChatPanel.append_reasoning_delta`. The GUI entry point is a checkable action-label button in the `ChatPanel` header (next to ＋新对话/历史会话) whose text flips **💭 思考可见**↔**思考不可见** to name the action offered; `_show_thinking` defaults False. **When off, reasoning_delta is dropped outright — no hidden widgets are built** (a max-effort CoT is huge). When on, each segment's thinking renders in an `InlineThinking` widget — a read-only `QPlainTextEdit` "sliding window" auto-sizing **both** width (≤720px) and height (≤160px) by counting wrapped lines from font metrics (`_sync_size_to_text`; **not** `documentSize`, stale after a width change), streaming via `insertPlainText` (O(delta) — `_saturated` skips the re-measure once both caps hit) and auto-following via `ensureCursorVisible()`; `mark_done()` stops the follow and retitles 思考中…→思考过程. The window must finalize **before** its tool row, so finalization fires from `_finalize_streaming` (via `finalize_assistant_segment`/`reset_streaming`) and the first answer token, not only the `final` event — else a tool-only iteration (no `assistant_delta`) locks in late. To bound the window *stack* (one per segment in a multi-tool turn), `append_reasoning_delta` collapses every earlier window when a new segment starts; only the active one stays expanded (a single-answer turn never triggers this). `load_history` replays each assistant message's persisted `reasoning_content` (when on) as a finalized window before its bubble. Offline `EchoLLMProvider`/`thinking=False` emit no reasoning, so none shows. `tests/test_chat_panel_thinking.py` covers orderings (off→none, answer→[thinking,bubble], tool-only→[thinking,tool]) plus toggle hide/reveal and history replay.

**macOS Calendar DDL notifications**: `CalendarReminderTool` (`src/tools/calendar_reminder.py`) adds a native-notification path — it shells to `osascript` to write selected assignment deadlines into a dedicated **PKU Captain** Calendar.app calendar (create-if-missing) as events with a `display alarm`, so macOS fires a real reminder ahead of each due time. It is macOS-gated (graceful `ToolResult` error elsewhere), surfaces the TCC Automation-denied case (`-1743`) as an actionable Chinese message (and short-circuits the rest of the batch on denial), and registers **online-only** (subprocess + online-only data). AppleScript gotchas baked into the script: pass event fields as `osascript -` argv (no shell/CJK escaping), build the date from numeric components (set `day` to 1 first to dodge month-overflow), and put `end date` in the `make new event` properties (Calendar saves at creation time; a later `set end date` fails with `-10025`). The dashboard entrypoint is a 加入日历 button on the 近期 DDL card (`AssignmentTodoCard`) that opens `CalendarReminderDialog` — a checkbox list of future, deadline-bearing assignments (seeded from the card's already-loaded data, no refetch) with an alarm-offset dropdown; the add runs through `run_async` (osascript blocks). The button gates on `calendar_reminder` registry membership like the other online-only surfaces. `tests/test_calendar_reminder_tool.py` mocks `subprocess.run` + `sys.platform` so it is deterministic on non-macOS CI and never touches the real Calendar.

**Multi-session chat** persists + reopens, driven by a **＋新对话** / **历史会话** button pair in the `ChatPanel` header (the latter opens `SessionHistoryDialog`, a modal open/delete list). `src/core/session_store.py` `SessionStore` (mirrors `ReminderStore`) writes one `data/sessions/<id>.json` per session (full history incl. `tool_calls` + `reasoning_content`, title, timestamps) on every turn-finish and on close — but **only after the first real user turn**, so an untouched window leaves no junk file. `src/core/session_titler.py` `SessionTitler` auto-names a session from its opening exchange via a lightweight `deepseek-v4-flash` provider in **non-think mode**, run async off the GUI thread (`run_async`); offline or key-less → a network-free heuristic title (`SessionTitler.heuristic`, never `EchoLLMProvider`/"echo: …"). ## Quickstart

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"
python -m src                  # PyQt6 window — defaults to offline (Echo LLM, offline tool subset)
python -m src --online         # PyQt6 window — DeepSeek + live tools (RAG off); needs secrets/api_keys/deepseek_key.txt
python -m src --online --rag    # also enable RAG knowledge_search; needs secrets/api_keys/embedding_key.txt
python -m src.cli --offline    # interactive REPL (no API key; EchoLLMProvider)
python -m src.cli              # interactive REPL against the real DeepSeek API
find src -name '*.py' -print0 | xargs -0 python -m py_compile  # syntax check
ruff check src                 # lint
pytest tests/                  # unit tests
python scripts/smoke_deepseek.py  # real DeepSeek round-trip — mandatory after core/llm/wire-format changes
```

The GUI defaults to offline so first-time clones don't hit the network. Pass `--online` once `secrets/api_keys/deepseek_key.txt` and the `pku3b` fork are in place. RAG `knowledge_search` is **off by default** even online — add `--rag` (and provide `secrets/api_keys/embedding_key.txt`) to register it. With `--rag` off there are no embedding-API calls at all.

### Worktrees (`claude --worktree`)

`.worktreeinclude` (gitignored, per-machine) copies only *gitignored* paths git won't carry over (tracked files like `CLAUDE.md` come via git). It covers `secrets/` (API keys + P-Lib/treehole creds + session) and `.local/` (`pku3b` binary), plus `.claude/*` + `!.claude/worktrees/` so a worktree gets local Claude config but not the 24 MB worktrees dir — use `.claude/*` not `.claude/`, else git prunes the dir before the `!`.

`.venv/` is **not** copied (~1.2 GB per worktree). Worktrees share the main venv via a symlink — run once inside the worktree (its `.claude/worktrees/<name>/` location puts the repo root three levels up):

```bash
ln -s ../../../.venv .venv
```

The shared venv's editable pointer (`_editable_impl_pku_captain.pth`) names the **main** checkout, so a bare `import src` from a worktree would resolve to main. The repo-root `conftest.py` prevents that — it front-loads the session's own root before any test imports `src`, so `pytest tests/` imports the worktree's `src/` (verified); `python -m src` / `scripts/smoke_deepseek.py` already front-load it (via `-m` / their own `sys.path.insert`). Don't `pip install` new deps from a worktree — it mutates the shared venv, leaking to main + siblings, so a dependency-adding branch can't be tested in isolation (install in the main checkout). The `.gitignore` entry is `.venv` (no trailing slash) so the symlink is ignored too.

Inside the REPL, `/save [path]` dumps the conversation (incl. `reasoning_content`) to `debug/conversation-<ts>.json` — useful for chasing thinking-mode wire-format regressions, since it preserves the exact assistant→tool→assistant ordering the DeepSeek endpoint validates. `debug/` is gitignored.

`pku3b` is an external binary; we ship against **our fork** at https://github.com/RizzoHou/pku3b (branch `master`, adds `--format json` to `assignment list` + `identity --format json`) so the Python wrapper consumes structured JSON instead of regex-parsing ANSI text. Install: `cargo install --git https://github.com/RizzoHou/pku3b --branch master` (needs Rust + pkg-config + libssl-dev) → `~/.cargo/bin/pku3b`; the wrapper resolves PATH first, then a repo-local `.local/cargo/bin/pku3b` fallback (`src/tools/pku3b.py`). Full procedure in `docs/setup_zh.md`. Don't roll back to upstream — the wrapper requires `--format json` and falls over without it. Quirk: pku3b stdout fails (exit 1, empty) when redirected to a file (`> x.json`, a known compio polling-backend bug); pipes work (`subprocess.run(capture_output=True)` uses them, so the wrapper is unaffected), so sanity-check via `pku3b ... | jq .`, not `>`.

The DeepSeek API key lives in `secrets/api_keys/deepseek_key.txt` (the whole `secrets/` directory is gitignored). Secrets follow a structured layout — `secrets/api_keys/{deepseek,embedding,kimi}_key.txt` (key files; the flat `secrets/<provider>_key.txt` form is still read as a legacy fallback), `secrets/plib/{email,password}` (P-Lib account — `PLibMaterialsTool` auto-injects these as `PLIB_EMAIL`/`PLIB_PASSWORD` so search/quota/download need no manual login), and `secrets/treehole/{id,password}` (treehole IAAA login; first online run prompts for SMS verification, then caches `session.json`). The `kimi_key.txt` is load-ready but no agent path consumes `KimiProvider` yet (vision feature is future scope). `DeepSeekProvider` defaults to model `deepseek-v4-pro` with reasoning `effort=max` — these are deliberate; don't substitute `deepseek-chat` or weaken the effort even if a stale model error tempts you to "fix" it, confirm with the captain first. The effort is sent on the wire as the `reasoning_effort` body key (verified live 2026-06-05: a top-level `effort` key is **silently ignored** — an invalid value still returns 200 — so the field name must stay `reasoning_effort` or `max` quietly drops back to the default `high`; valid values are `high`/`low`/`medium`/`max`/`xhigh`). The thinking-mode endpoint **requires** every previous assistant turn's `reasoning_content` to be replayed on the next request (otherwise: `400 invalid_request_error`); this is wired through `ChatResponse → ChatMessage.reasoning_content → DeepSeekProvider._to_api_message`. Preserve that round-trip when touching any of the three. The lighter `deepseek-v4-flash` channel (session auto-naming only) runs **non-think mode**: `DeepSeekProvider(thinking=False)` drops `reasoning_effort` and sends `{"thinking": {"type": "disabled"}}` in the body, returning plain `content` with no `reasoning_content` to replay. `thinking=True` is the default and is used everywhere else — that path's request body is byte-identical to before the param existed (now keyed `reasoning_effort`), so don't "simplify" the `_build_body` branch.

## What this project is

**PKU Captain** is a desktop AI assistant for PKU students, submitted as a course assignment for an OOP / programming-practice class. Three-person team. The product is a PyQt6 desktop app whose **end-state center of gravity is the dashboard** — a single-screen "信息总站" surfacing today's classes, near-term DDLs, course notices and lectures, with an extensible `Source` registry meant to absorb more PKU information sources over time (教务网 / 树洞 / 公众号 as post-06-06 additions). A side-panel **LLM agent** handles natural-language questions about that same data (via `pku3b`, lectures, etc.) and orchestrates multi-step workflows. Note: Week 1 is built **agent-first** as a vertical slice to prove the four OOP hierarchies — that's a build-order choice, not the end-state UX. See `docs/design_reference_zh.md` for the full vision and `docs/roadmap_zh.md` "产品定位（终态）" for the dashboard-vs-agent ordering call.
