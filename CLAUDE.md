The same PR collapsed the GUI to a **2-panel** layout (`dashboard | chat`, tool calls now inline in the chat flow via `InlineToolCall`; `ToolTracePanel` is retained-but-unmounted) and added dashboard dialogs for treehole/P-Lib/announcements/lectures/reminders/memory/knowledge. **Dashboard dialogs follow integration-contract §1+§5**: they obtain tools from the injected `ToolRegistry` (`tools.find(name)`, `name in tools`) — never by constructing `Tool` subclasses — with online-only entries gated on registry membership (so offline never touches a network/subprocess tool), and every blocking dialog call goes through `src/ui/tool_call_worker.py`'s `run_async()` onto a `QThreadPool` to keep the window responsive. **Dashboard refresh is scoped, not global**: the header 刷新 button fires the panel's `refresh_requested` (reloads all cards), but each card's own 刷新 button (and a post-login `auth_changed`) must emit `partial_refresh_requested([tool_key])` so `DashboardWorker.refresh(keys=...)` reloads only that card — wiring a card's refresh to the global `refresh_requested` reintroduces the "one click refreshes everything" bug. **Dashboard refresh is also parallel, not serial**: `DashboardWorker.refresh` fans the selected tools out over a `ThreadPoolExecutor` and emits each card's result via `as_completed` from the worker thread (signal wiring + `_busy`/`finished` semantics unchanged), so startup time is the slowest single card, not the sum — the cards block on subprocess/network I/O so this is a real win. Concurrent refresh is safe by **error isolation**, not by an audited guarantee: `_invoke` swallows every exception and each card reports independently via `item_error`, so the worst a concurrency mishap can do is make one card show an error a re-refresh fixes — it can't crash the dashboard or corrupt a sibling card. The one shared resource within a single refresh is pku3b's content-addressed cache: on a cold/stale session `pku3b_assignments` + `pku3b_announcements` both target the `Blackboard::_get_courses` cache key and may each fetch + re-login (the worktree author's claim, from reading the Rust fork source — not re-verified here), worst case a redundant fetch; `cfg.toml` is read-only on these paths, so no credential race. The Python-side creds injection is concurrency-safe regardless — `PLibMaterialsTool` builds env per call via `os.environ.copy()` + `env=`, never mutating the global. `tests/test_dashboard_worker_refresh.py` guards this with a `threading.Barrier` test that a serial loop cannot pass. **Chat renders one bubble per assistant segment, not per turn**: a tool-using turn runs multiple LLM iterations, each emitting an assistant text segment then tool calls, so `MainWindow._on_agent_event` calls `ChatPanel.finalize_assistant_segment()` on every `tool_call` to lock the current streaming bubble in place before the tool row — without it, later segments overwrite the first bubble (the `final` event's `add_assistant_message` rewrites it) and the last reply renders above its tool calls instead of below; `tests/test_chat_panel_segments.py` drives real `AgentEvent`s through `_on_agent_event` (offscreen Qt) to guard the ordering. Keep the "one offline reference, real ones for teammates" pattern when adding more scaffolding.

## Quickstart

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"
python -m src                  # PyQt6 window — defaults to offline (Echo LLM, offline tool subset)
python -m src --online         # PyQt6 window — DeepSeek + live tools (RAG off); needs secrets/api_keys/deepseek_key.txt
python -m src --online --rag    # also enable RAG knowledge_search; needs secrets/api_keys/embedding_key.txt
python -m src.cli --offline    # interactive REPL (no API key; EchoLLMProvider)
python -m src.cli              # interactive REPL against the real DeepSeek API
find src -name '*.py' -print0 | xargs -0 python -m py_compile  # syntax check
ruff check src                 # lint
pytest tests/                  # unit tests
python scripts/smoke_deepseek.py  # real DeepSeek round-trip — mandatory after core/llm/wire-format changes
```

The GUI defaults to offline so first-time clones don't hit the network. Pass `--online` once `secrets/api_keys/deepseek_key.txt` and the `pku3b` fork are in place. RAG `knowledge_search` is **off by default** even online — add `--rag` (and provide `secrets/api_keys/embedding_key.txt`) to register it. With `--rag` off there are no embedding-API calls at all.

### Worktrees (`claude --worktree`)

`CLAUDE.md` is tracked, so every worktree already has it via git — listing it in `.worktreeinclude` would be a no-op (that file copies only *gitignored* files). A repo-root `.worktreeinclude` (gitignored) covers the paths git won't carry over: `secrets/` (API keys + P-Lib/treehole creds + cached treehole session) and `.local/` (the project-local `pku3b` cargo binary).

`.venv/` is **not** copied (~1.2 GB per worktree). Worktrees share the main venv via a symlink — run once inside the worktree (its `.claude/worktrees/<name>/` location puts the repo root three levels up):

```bash
ln -s ../../../.venv .venv
```

The shared venv's editable pointer (`_editable_impl_pku_captain.pth`) names the **main** checkout, so on its own `import src` from a worktree would resolve to main. The repo-root `conftest.py` prevents that — pytest loads it before any test imports `src` and it front-loads the session's own root, so bare `pytest tests/` imports the worktree's `src/` (verified). `python -m src` and `python scripts/smoke_deepseek.py` already front-load the worktree root (via `-m` / their own `sys.path.insert`). One don't: don't `pip install` new deps from inside a worktree — it mutates the shared venv, leaking to main + sibling worktrees, so a branch that *adds* a dependency can't be tested in isolation (install it in the main checkout instead). The `.gitignore` entry is `.venv` (no trailing slash) so the symlink is ignored too.

Inside the REPL, `/save [path]` dumps the conversation (including `reasoning_content`) to `debug/conversation-<ts>.json` by default — useful when chasing thinking-mode wire-format regressions, since the dump preserves the exact assistant→tool→assistant ordering the DeepSeek endpoint validates against. `debug/` is gitignored.

`pku3b` is installed at `/home/ubuntu/.local/bin/pku3b` on this machine — but we ship against **our fork** at https://github.com/RizzoHou/pku3b (branch `feat/assignment-list-json-output`), which adds `--format json` to `assignment list` so the Python wrapper consumes structured data instead of regex-parsing rendered ANSI text. The patched binary at `~/.local/bin/pku3b` is v0.13.0+ from that branch. Teammates install with `cargo install --git https://github.com/RizzoHou/pku3b --branch feat/assignment-list-json-output` (requires Rust + pkg-config + libssl-dev). The team-facing copy of this install procedure lives in `docs/setup_zh.md`. Source checkout lives at `~/projects/pku3b/` (separate from this repo). Don't roll back to upstream — the Python wrapper now requires `--format json` and falls over without it. One quirk worth flagging: pku3b's stdout fails (exit 1, empty output) when redirected to a regular file (`pku3b ... > /tmp/x.json`) — it's a known upstream issue with compio's polling backend. Pipes work, which is what `subprocess.run(capture_output=True)` uses, so the Python wrapper is unaffected; but if you're sanity-checking the CLI from a shell, use a pipe (`pku3b ... | jq .`) instead of `>`.

The DeepSeek API key lives in `secrets/api_keys/deepseek_key.txt` (the whole `secrets/` directory is gitignored). Secrets follow a structured layout — `secrets/api_keys/{deepseek,embedding,kimi}_key.txt` (key files; the flat `secrets/<provider>_key.txt` form is still read as a legacy fallback), `secrets/plib/{email,password}` (P-Lib account — `PLibMaterialsTool` auto-injects these as `PLIB_EMAIL`/`PLIB_PASSWORD` so search/quota/download need no manual login), and `secrets/treehole/{id,password}` (treehole IAAA login; first online run prompts for SMS verification, then caches `session.json`). The `kimi_key.txt` is load-ready but no agent path consumes `KimiProvider` yet (vision feature is future scope). `DeepSeekProvider` defaults to model `deepseek-v4-pro` with reasoning `effort=max` — these are deliberate; don't substitute `deepseek-chat` or strip `effort` even if a stale model error tempts you to "fix" it, confirm with the captain first. The thinking-mode endpoint **requires** every previous assistant turn's `reasoning_content` to be replayed on the next request (otherwise: `400 invalid_request_error`); this is wired through `ChatResponse → ChatMessage.reasoning_content → DeepSeekProvider._to_api_message`. Preserve that round-trip when touching any of the three.

## What this project is

**PKU Captain** is a desktop AI assistant for PKU students, submitted as a course assignment for an OOP / programming-practice class. Three-person team. The product is a PyQt6 desktop app whose **end-state center of gravity is the dashboard** — a single-screen "信息总站" surfacing today's classes, near-term DDLs, course notices and lectures, with an extensible `Source` registry meant to absorb more PKU information sources over time (教务网 / 树洞 / 公众号 as post-06-06 additions). A side-panel **LLM agent** handles natural-language questions about that same data (via `pku3b`, lectures, etc.) and orchestrates multi-step workflows. Note: Week 1 is built **agent-first** as a vertical slice to prove the four OOP hierarchies — that's a build-order choice, not the end-state UX. See `docs/design_reference_zh.md` for the full vision and `docs/roadmap_zh.md` "产品定位（终态）" for the dashboard-vs-agent ordering call.

## When you write code here

- **Stack** is now committed: Python 3.11+ / PyQt6 / DeepSeek (chat) + Kimi (vision) via `LLMProvider` abstraction / DashScope `text-embedding-v4` embeddings via API (no local model) / SQLite + numpy for vectors / `pku3b` Rust CLI as subprocess.
- **OOP is the point of the course.** New tools, workflows, LLMs, or sources go in as subclasses of the existing ABCs in `src/{tools,workflows,llm}/base.py` and `src/rag/source.py`, and register against the matching `*Registry` at the call site (keep modules side-effect-free — no auto-registration at import). Resist ad-hoc dispatch — the "subclass + register" pattern is the rubric showcase.
- **Default to English** — interaction, operational docs (`docs/tasks/`), commit messages, in-code comments, code identifiers. **Chinese** is reserved for end-user-facing product copy (the app's users are PKU students) and the existing `*_zh.md` design docs (`design_reference_zh`, `roadmap_zh`, `schedule_zh`, `integration_contract_zh`, `setup_zh`) — don't translate those unprompted.
