## Repo status

The app runs end-to-end (live GUI + tools, smoke test green). `src/` is the importable Python package (intra-package relative imports, so `src` rarely shows in code); **the component layout, the four OOP hierarchies, and the runtime data flows now live in `ARCHITECTURE.md`** — this section keeps the rules/invariants, not the structural map. `pyproject.toml` builds via hatchling (`packages = ["src"]`); runtime deps PyQt6/requests/numpy (embeddings are a hosted API — no torch/transformers stack), dev deps pytest/ruff/mypy. The pytest suite is small but real (`tests/`, run `pytest tests/`). `scripts/smoke_deepseek.py` drives a real DeepSeek round-trip through the agent loop and is **mandatory after any change to `core/`, `llm/`, or the tool-call wire format**. `src/cli.py` (`python -m src.cli`, `--offline` / `--show-reasoning`) is the matching REPL on the same loop — the contract-conformance probe for the GUI seam because it goes through `build_agent()`. One reference offline subclass per hierarchy (`ClockTool`, `EchoLLMProvider`, `StaticSource`, `HelloWorkflow`) lets the loop + tests run without API keys or live PKU endpoints. `src/tools/pku3b.py` is the shared subprocess wrapper (locates the binary, timeout, ANSI-strips stderr — **not** itself a Tool subclass); `PKU3bAssignmentsTool` consumes `pku3b assignment list --format json` and `json.loads`es it directly (no regex), surfacing course/assignment ids, raw + ISO-8601 deadlines, completion flag, descriptions, and Blackboard attachment URIs.

The `src/rag` embedding chain (`Embedder`/`KnowledgeBase`/`KnowledgeSearchTool`, over DashScope `text-embedding-v4`) is **superseded by the doc base and no longer registered** — kept as an OOP artifact. `DeanSource`/`CalendarSource` wire through `build_source_registry()`; `MemoryTool` is the persistent preference store under `data/` (`MemoryStore` backend in `src/core/memory.py`).

**Memory is folded into responses, not only tool-reachable**: bootstrap shares **one** `MemoryStore` instance between `MemoryTool(store=...)` and `Agent(memory=...)`, and each `Agent.turn()` iteration merges the current entries — via `render_memory_context()` — into the *leading* system message of a snapshot copy. It is never appended as a second system turn (DeepSeek expects one) and never written into `Conversation`, so the block refreshes after a mid-turn `set` yet rendered history stays clean. Two invariants when touching this: keep the store **shared** — a second `MemoryStore` on the same path silently no-ops because it loads once at construction and never re-reads — and keep the block out of `Conversation`.

**Natural-language input becomes memory without a manual key**: `MemoryStore.remember(text)` stores free text under an auto-derived content-hash handle (`note-<sha1>`) — the keyless producer of the *same* entry kind as `set` (no `note`/preference split; `render_memory_context` stays uniform), idempotent on identical text, no semantic overwrite. `MemoryTool` exposes it as the `remember` action; the system prompt tells the agent to prefer `remember` for facts learned mid-conversation (name, location, language, schedule) and reserve `set` for replacing a value under a known key. The dashboard `MemoryDialog` accordingly dropped its key+value form for a single natural-text box → 记住 button (signature stays `(tool, parent)` plus an additive `learner=` — `test_dashboard_dialogs.py` constructs it positionally).

**The dashboard box also does smart LLM extraction**: `MemoryLearnService(llm, store)` (`src/core/memory_learn.py`, exported from `src.core`) is a dedicated *stateless* extraction — prompts the LLM for a JSON array of atomic facts and stores each via `remember` — **not** a detour through `Agent.turn()` (no chat-history pollution / event noise, same result). `MainWindow` builds it from the agent's **own** `llm` + `memory` (shared store, so dashboard- and chat-learned facts coincide) and injects it into `DashboardPanel(memory_learner=...)` → `MemoryDialog(learner=...)`; with a learner the 记住 button runs the split through `run_async` (LLM call) and shows "已记住 N 条", without one (offline) it stores verbatim inline. The JSON parser (`_parse_facts`) **is the feature** — it must tolerate ```code fences``` and prose, or every real reply silently degrades to one verbatim blob; offline `EchoLLMProvider` never yields JSON so it falls back to verbatim with no special-casing. Tradeoff: free-text notes don't overwrite (delete the stale one from the list). Tests: `test_memory_{store,tool,agent}.py` — the last proves a learned fact folds forward into the next call and that `build_agent` shares one store), `PKU3bAnnouncementsTool` (no dates in `announcement list` → opt-in `resolve_dates`, **dashboard-only**, caches dates in `data/announcement_dates.json`; prewarm fresh checkouts, undated→历史-only); and `KimiProvider` (Kimi K2.6 — a switchable chat brain + vision; see the LLM section).

Network tools register **online only** in `bootstrap._build_tools()` because they touch a subprocess or the network.

The **doc base** replaces RAG (`src/tools/doc_base.py`): `doc_search` — lexical ranked search over `doc_base/manifest.json` (title/breadcrumb only; empty query browses all) — registers in **every** mode (no index/network). `doc_read` renders a doc's PDF pages via `pdftoppm` and returns them as **`ToolResult.images`**; `Agent.turn()` injects them as a multimodal `user` message (`Conversation.add_user_parts`) **after** all tool results, so the **Kimi K2.6 chat brain reads the pages itself** — no PDF→text middleman. `doc_read` registers **only while the active brain is Kimi** (text brains can't read images), toggled by `apply_chat_model`; caps 24 pages, `pages` slices longer docs. The dashboard 树洞消息 card **stays in sync with that auto-checking**. Two parts.

(1) **Accumulate, don't replace**: `monitor.check()` returns only the delta since the dashboard's **own** cursor (`secrets/treehole/state.json` — deliberately **separate** from the notifier's `data/treehole-notify-state.json`; concurrent writes are benign), so replacing per-poll made a reply **vanish on the next empty poll**. Instead a persisted `TreeholeInboxStore` + pure `merge_treehole_updates` (both in `treehole_updates.py`) union by pid/cid into `data/treehole_inbox.json` and keep unread visible until the user opens `TreeholeMessagesDialog` (mark-as-read → `clear`); the inbox is **in-memory by default** (`path=None`, so tests + offline never touch `data/`), injected into `DashboardPanel(...)` by `MainWindow`. `set_treehole_updates` / `set_error` merge-and-render-from-inbox rather than overwrite, so a transient poll error keeps the accumulated rows + badge. Sibling **`TreeholeHistoryStore`** (`data/treehole_history.json`, append-only, per-comment by (pid,cid)) takes the **same raw poll** but is **never cleared on read** — backs the dialog's 历史消息 tab (newest-first, + 清空历史); feed from the poll, not the post-clear inbox (count-only polls / per-poll `limit` are gaps).

(2) **Auto-refresh on the notifier's cadence**: a `MainWindow` `QTimer` re-polls the card while notifications are enabled+supported+**logged-in** (`_reconfigure_treehole_sync`, re-read on startup + on `treehole_settings_changed` when the dialog closes), via a **quiet**, busy-guarded `run_async` poll (no flicker/spam). Limits: the card re-derives via its *own* poll, not the daemon's Notification-Center list (count-based trigger can drift); app-closed replies still surface on reopen because the cursor is frozen while shut. **Don't** unify the (now three) state files or revert accumulation to replace (re-introduces the two-poller race / vanish-on-poll bugs). Tests: `test_treehole_inbox.py` / `test_treehole_history.py` (stores) + `test_treehole_sync.py` (accumulate / mark-read / history-survives-clear / error-preserve, timer gating).

Tool calls render inline via `InlineToolCall` (`ToolTracePanel` retained-but-unmounted); dashboard dialogs for treehole/P-Lib/announcements/memory/文档库.

**Dashboard dialogs follow integration-contract §1+§5**: they obtain tools from the injected `ToolRegistry` (`tools.find(name)`, `name in tools`) — never by constructing `Tool` subclasses — with online-only entries gated on registry membership (so offline never touches a network/subprocess tool), and every blocking dialog call goes through `src/ui/tool_call_worker.py`'s `run_async()` onto a `QThreadPool` to keep the window responsive.

**Dashboard refresh is scoped, not global**: the header 刷新 button fires the panel's `refresh_requested` (reloads all cards), but each card's own 刷新 button (and a post-login `auth_changed`) must emit `partial_refresh_requested([tool_key])` so `DashboardWorker.refresh(keys=...)` reloads only that card — wiring a card's refresh to the global `refresh_requested` reintroduces the "one click refreshes everything" bug.

**Dashboard refresh is also parallel, not serial**: `DashboardWorker.refresh` fans the selected tools out over a `ThreadPoolExecutor` and emits each card's result via `as_completed` from the worker thread, so startup time is the slowest single card, not the sum. Concurrent refresh is safe by **error isolation**: `_invoke` swallows every exception and each card reports independently via `item_error`, so one card's failure never crashes the dashboard or corrupts a sibling. The one shared resource within a refresh is pku3b's content-addressed cache: on a cold/stale session `pku3b_assignments` + `pku3b_announcements` share the `Blackboard::_get_courses` key and may each fetch + re-login, worst case a redundant refetch; `cfg.toml` is read-only here, so no credential race. Creds injection is concurrency-safe too — `PLibMaterialsTool` builds env per call (`os.environ.copy()` + `env=`), never mutating the global. `tests/test_dashboard_worker_refresh.py` guards this with a `threading.Barrier` test a serial loop cannot pass.

**Dashboard cards are cached to disk for instant reload**: `DashboardCache` (`src/core/dashboard_cache.py`, built via `build_dashboard_cache()`, lazily re-exported) saves each card's **raw tool payload** to `data/dashboard_cache/<tool_key>.json` — one dumb, lock-guarded file per card, mirroring `SessionStore` (no in-memory mirror; the GUI owns change-detection). `MainWindow` paints from `load_all()` at startup (`_seed_dashboard_from_cache`), then fires a **silent** refresh (`_start_refresh([], silent=True)`) that suppresses the spinner on already-cached cards and repaints only the ones whose data changed — a canonical-JSON `_signature` compare, with `_should_render` enforcing the invariant that a card put into the loading state (tracked in `_loading_keys`) always renders so it can never strand on the spinner. Explicit header/per-card refresh stays non-silent (always spins + repaints, for click feedback). A refresh error keeps the last-good cached card visible with a status-bar note rather than wiping it to an error — so an offline launch shows the prior online cache (deliberate; the `离线模式` subtitle signals it). Caching **raw** payloads (not rendered widgets) means time-relative renders like `upcoming_assignments` re-filter against today's clock on repaint. `tests/test_dashboard_cache.py` covers the store plus the pure `_signature` / `_should_render` decision (full MainWindow wiring validated headless, not committed).

**Chat renders one bubble per assistant segment, not per turn**: a tool-using turn runs multiple LLM iterations, each emitting an assistant text segment then tool calls, so `MainWindow._on_agent_event` calls `ChatPanel.finalize_assistant_segment()` on every `tool_call` to lock the current streaming bubble in place before the tool row — without it, later segments overwrite the first bubble (the `final` event's `add_assistant_message` rewrites it) and the last reply renders above its tool calls instead of below; `tests/test_chat_panel_segments.py` drives real `AgentEvent`s through `_on_agent_event` (offscreen Qt) to guard the ordering.

**Chain-of-thought ("thinking") is optionally visible, off by default**: `DeepSeekProvider.stream_chat` yields `ChatStreamEvent(reasoning_delta=...)` per token (+ accumulating `reasoning_content` for replay); `Agent.turn()` forwards an additive `reasoning_delta` AgentEvent (contract §3; before `assistant_delta` per segment) → `ChatPanel.append_reasoning_delta`. A checkable `ChatPanel` header button flips **💭 思考可见**↔**思考不可见** (`_show_thinking` defaults False). **When off, reasoning_delta is dropped outright — no hidden widgets are built.** When on, each segment renders in an `InlineThinking` widget — a read-only sliding-window `QPlainTextEdit` that auto-sizes width/height from font metrics, measured **after `ensurePolished()`** so the late-applied `#InlineThinkingBody` QSS font is used (else the first window renders too tall), **not** `documentSize` (stale after a width change). Two ordering rules: the window must **finalize before its tool row** — finalization fires from `_finalize_streaming` and the first answer token, not only the `final` event, else a tool-only iteration locks in late — and `append_reasoning_delta` **collapses earlier windows** when a new segment starts (bounds the per-segment stack). `load_history` replays persisted `reasoning_content` (when on) as a finalized window before its bubble; offline `EchoLLMProvider`/`thinking=False` emit none. `tests/test_chat_panel_thinking.py` covers orderings, toggle, history replay, first-window auto-fit.

**macOS Calendar DDL notifications**: `CalendarReminderTool` (`src/tools/calendar_reminder.py`) shells to `osascript` to write selected assignment deadlines into a dedicated **PKU Captain** Calendar.app calendar (create-if-missing) as events with a `display alarm`. macOS-gated (graceful `ToolResult` error elsewhere), surfaces TCC Automation-denied (`-1743`) as an actionable Chinese message (and short-circuits the batch on denial), registers **online-only**. AppleScript gotchas in the script: pass fields as `osascript -` argv (no shell/CJK escaping); build the date from numeric components (set `day` to 1 first to dodge month-overflow); put `end date` in the `make new event` properties (a later `set end date` fails `-10025`). Dashboard: a 加入日历 button on the 近期 DDL card (`AssignmentTodoCard`) opens `CalendarReminderDialog` (future deadline-bearing assignments seeded from the card, alarm-offset dropdown; add via `run_async`), gated on `calendar_reminder` membership. `tests/test_calendar_reminder_tool.py` mocks `subprocess.run` + `sys.platform`.

**Multi-session chat** persists + reopens via a **＋新对话** / **历史会话** header pair (`SessionHistoryDialog` = modal open/delete list). `src/core/session_store.py` `SessionStore` writes one `data/sessions/<id>.json` (full history incl. `tool_calls` + `reasoning_content`, title, timestamps) per turn-finish and on close — but **only after the first real user turn**, so an untouched window leaves no junk file. `src/core/session_titler.py` `SessionTitler` auto-names from the opening exchange via a `deepseek-v4-flash` provider in **non-think mode**, async (`run_async`); offline/key-less → a network-free `SessionTitler.heuristic` (never `EchoLLMProvider`). ### Worktrees (`claude --worktree`)

`.worktreeinclude` (gitignored, per-machine) copies only *gitignored* paths git won't carry over (tracked files like `CLAUDE.md` come via git). It covers `secrets/` (API keys + P-Lib/treehole creds + session) and `.local/` (`pku3b` binary), plus `.claude/*` + `!.claude/worktrees/` so a worktree gets local Claude config but not the 24 MB worktrees dir — use `.claude/*` not `.claude/`, else git prunes the dir before the `!`.

`.venv/` is **not** copied (~1.2 GB per worktree). Worktrees share the main venv via a symlink — run once inside the worktree (its `.claude/worktrees/<name>/` location puts the repo root three levels up):

```bash
ln -s ../../../.venv .venv
```

The shared venv's editable pointer names the **main** checkout, so a bare `import src` from a worktree resolves to main; the repo-root `conftest.py` front-loads the session's own root first so `pytest tests/` imports the worktree's `src/` (`python -m src` / `scripts/smoke_deepseek.py` already front-load via `-m` / `sys.path.insert`). Don't `pip install` from a worktree — it mutates the shared venv (leaks to main + siblings); install in the main checkout. `.gitignore` is `.venv` (no slash) so the symlink is ignored too.

REPL `/save [path]` dumps the conversation (incl. `reasoning_content`) to `debug/conversation-<ts>.json` (gitignored) — handy for thinking-mode wire-format regressions (preserves the exact assistant→tool→assistant ordering the endpoint validates).

`pku3b` is an external binary; we ship against **our fork** at https://github.com/RizzoHou/pku3b (branch `master`, adds `--format json` to `assignment list` + `identity --format json`) so the Python wrapper consumes structured JSON instead of regex-parsing ANSI text. Install per `docs/setup_zh.md` → `~/.cargo/bin/pku3b`; the wrapper resolves PATH first, then a repo-local `.local/cargo/bin/pku3b` fallback (`src/tools/pku3b.py`). Don't roll back to upstream — the wrapper requires `--format json`. Quirk: pku3b stdout fails when redirected to a file (`> x.json`, a compio bug); pipes work (so `capture_output=True` is fine) — debug via `| jq .`, not `>`.

The DeepSeek API key lives in `secrets/api_keys/deepseek_key.txt` (the whole `secrets/` directory is gitignored). Secrets layout — `secrets/api_keys/{deepseek,kimi}_key.txt` (flat `secrets/<provider>_key.txt` is a legacy fallback), `secrets/plib/{email,password}` (`PLibMaterialsTool` auto-injects `PLIB_EMAIL`/`PLIB_PASSWORD`), `secrets/treehole/{id,password}` (IAAA; first run SMS-verifies then caches `session.json`) — full notes in `docs/setup_zh.md`. **`KimiProvider` is a full chat brain** (`kimi-k2.6`, 256k, `api.moonshot.cn/v1`) mirroring DeepSeek — SSE `stream_chat`, thinking via `{"thinking":{"type":"enabled"}}`→`reasoning_content` (replay **optional**, unlike DeepSeek), tools, usage. The chat header switches brain DeepSeek⇄Kimi (`src.core`: `available_chat_models`/`build_chat_llm`/`apply_chat_model`/`DEFAULT_CHAT_MODEL`); a switch **resets the chat (reset-on-switch)** so no conversation mixes models, gates `doc_read` on the brain, and the context meter auto-adapts via `llm.context_window` (1M⇄256k). `VisionRouter` auto-switches a 培养方案/文档 question on DeepSeek into a fresh Kimi chat; `session_store` collapses injected image `user` messages to a text label on persist. **Both providers' `stream_chat` iterate raw bytes then UTF-8-decode** — `iter_lines(decode_unicode=True)` corrupts multi-byte CJK `reasoning_content` at chunk boundaries (don't revert). `DeepSeekProvider` defaults to model `deepseek-v4-pro` with reasoning `effort=max` — these are deliberate; don't substitute `deepseek-chat` or weaken the effort even if a stale model error tempts you to "fix" it, confirm with the captain first. The effort is sent on the wire as the `reasoning_effort` body key (the field name must stay `reasoning_effort` — a top-level `effort` is silently ignored; valid: `high`/`low`/`medium`/`max`/`xhigh`). The thinking-mode endpoint **requires** every previous assistant turn's `reasoning_content` to be replayed on the next request (otherwise: `400 invalid_request_error`); this is wired through `ChatResponse → ChatMessage.reasoning_content → DeepSeekProvider._to_api_message`. Preserve that round-trip when touching any of the three. The `deepseek-v4-flash` channel (session auto-naming) runs **non-think**: `DeepSeekProvider(thinking=False)` drops `reasoning_effort`, sends `{"thinking":{"type":"disabled"}}`, returns plain `content`. `thinking=True` (default everywhere else) keeps the historical body — don't "simplify" the `_build_body` branch.

## What this project is

**PKU Captain** is a desktop AI assistant for PKU students, submitted as a course assignment for an OOP / programming-practice class. Three-person team. The product is a PyQt6 desktop app whose **end-state center of gravity is the dashboard** — a single-screen "信息总站" surfacing today's classes, near-term DDLs, and course notices, with an extensible `Source` registry meant to absorb more PKU information sources over time (教务网 / 树洞 / 公众号 as post-06-06 additions). A side-panel **LLM agent** handles natural-language questions about that same data (via `pku3b`, etc.) and orchestrates multi-step workflows. Note: Week 1 is built **agent-first** as a vertical slice to prove the four OOP hierarchies — that's a build-order choice, not the end-state UX. See `docs/design_reference_zh.md` for the full vision and `docs/roadmap_zh.md` "产品定位（终态）" for the dashboard-vs-agent ordering call.

## Agentic auditing workflow

Three repo-root artifacts keep the project auditable by a non-coding architect — verification relocated off the code onto stack-agnostic prose + observable behavior (seed: `docs/external/pku-captain-agentic-workflow-handoff.md`). Each is maintained by a same-named project-level skill in `.claude/skills/` (tracked via the `.gitignore` un-ignore, so teammates/worktrees inherit them); the SKILL.md files hold the detailed rules. Fire each **by change type**, not every task:

- `ARCHITECTURE.md` — structural map (skill `architecture`, structural changes only). *Structure* here; *rules* in "Repo status"; *seam* in `integration_contract_zh.md` — cross-reference, never restate.
- `DEVCHANGELOG.md` — decision log, why X over Y (skill `devchangelog`, ~every non-trivial change). *Why* here; *what-shipped* in `CHANGELOG.md`.
- `VERIFICATION.md` — human-run verification steps, Pending→Verified (skill `verification`, user-visible / release-critical / beyond-`pytest` changes).

Experimental machinery — to be packaged as a Claude plugin once mature.

**Plan-gate.** For any non-trivial/structural change, state the plan in plain prose *before* coding — files, approach, what could break, assumptions — and let the architect approve the **plan, not the diff**. Push boundary decisions (new service, new global) UP into the plan; leave idioms (which Qt signal, which API call) DOWN in code. Trivial changes (typo, one-liner, test tweak) skip it. Narrower than the `claude-md-improver` auto-apply waiver below, which covers only this file.

**Credentials** leak invisibly to behavior tests (the app works while leaking) — audit by hand before release; the live stderr→`ToolResult.error`→LLM-context risk and its checks are in `VERIFICATION.md` "Credential pre-release audit".
