本文件配合 `design_reference_zh.md` 与 `schedule_zh.md` 使用：前者定义"做什么"，后者定义"何时交付"，本文件定义"如何分阶段抵达"。

## 三个窗口

| 窗口 | 时间区间 | 天数 | 目标 |
| --- | --- | --- | --- |
| 演示前 | 2026-05-12 → 2026-06-06 | ~25 | 端到端可运行；6 项核心功能均以有实质深度的形态出现 |

## 产品定位（终态）

PKU Captain 的终态是一个面向 PKU 学生的 **信息总站**：仪表盘是主舞台，承载今日课表 / 近期 DDL / 通知 / 讲座 / 天气，并以可扩展的方式持续接入更多 PKU 信息源；对话 agent 以侧栏形式存在，承担多步骤工作流编排与自然语言查询，而非作为主入口。与设计文档"核心功能"列表一致（仪表盘 #1，对话侧栏 #2）。

**构建顺序 ≠ 终态布局**：第 1 周仍采用 agent-first 垂直切片，让 4 条 OOP 继承体系（Tool / Workflow / LLMProvider / Source）在一周内全部走通一次，并稳定产出一个 30 秒演示片段。从第 2 周起，重心切回仪表盘，agent 退回侧栏位置。

**可扩展性 ≠ 实现广度**：`Source` 注册表与 `Tool` 注册表本身就是 OOP 扩展承载体。06-06 前只需稳交付少量官方 `Source`（教务部、校历）；教务网（SSO 鉴权）、树洞（SSO + UGC + ToS 风险）、微信公众号（生态抓取脆性）这类高门槛源留到最终提交窗口，以新 `Source` 子类形式追加，作为"信息总站"招牌的延展。

## 阶段策略

- **垂直切片优先（构建顺序）**：第 1 周先打通"用户提问 → 工具调用 → 结果呈现"的最小端到端链路，让四条继承体系各跑通一次；之后再向两侧铺开仪表盘、工具与工作流。一旦进度滑坡，损失的是功能而非整体可演示性。
- **PR + squash 合并**；`main` 永远可演示。CODEOWNERS 已配置，PR 按目录自动指派评审者。
- **协作约定**：各周任务以 to-do 列表形式列出，不预先分配负责人；任何成员均可认领未勾选的条目。完成后将其勾选，并在条目末尾追加 `— @github_handle` 作为 involver；后续若有人评审、修改或补充已完成的条目，将自己的 GitHub 用户名追加到 involver 列表（多人以逗号分隔）。Captain 负责跨任务整合与评审协调。

## 削减顺序（约定）

当任意条目在周检查点落后 >2 天时，按以下顺序削减（captain 现场拍板，不再讨论）：

1. `KimiProvider`（视觉 LLM）— DeepSeek 单独足够支撑演示
2. `WeeklyReviewWorkflow`
3. `CourseCatchupWorkflow`
4. 记忆偏好编辑器 UI（保留后端持久化）
5. 第 3 个 RAG Source（至少保留 1 个）
6. 红点提示
7. `LectureTool`（保留 pku3b / 天气）
8. 仪表盘除"课表 / DDL / 通知"以外的小部件

**绝不削减**：Agent 内核及可见工具调用、`PKU3bAssignmentsTool`、含课表 / DDL / 通知的仪表盘、`MorningBriefingWorkflow`、基础记忆持久化、1 个 RAG Source + `KnowledgeSearchTool`。

## 路演缓冲窗口（6-07 至 6-12）

仅打磨。本窗口不合并任何新功能。任务：演示录像迭代；路演幻灯与现场脚本；首映反馈对应的 bug 修复；代码注释 / docstring 一次过；6-09 在演示机上做一次彩排。

## 最终提交窗口（6-13 至 7-06）

按 captain 选定的雄心上限分两段推进。

### 4-5 周（6-13 至 6-26）— 招牌特性

- **自动刷新 RAG**：每 Source 自身节奏的 Scheduler；SHA-256 差分；增量重嵌入；后台 async 循环；新内容红点。设计文档点名的招牌组件。
- **信息源扩展（"信息总站"招牌）**：以新 `Source` 子类形式接入更高门槛的源 — 候选：教务网（SSO 鉴权）、PKU 树洞（SSO + UGC，注意 ToS 与署名）、相关微信公众号（如教务部 OA 推文）。优先按 鉴权门槛低 → 高 与 ToS 风险低 → 高 排序逐个接入；任何源若 4-5 周内未稳定，仍按 `Source` 注册机制留作扩展点而非阻塞项。
- **高级工作流**：例如"考试周复盘"、"选课助手"；工作流组合原语。
- **工具插件化**：从 `plugins/` 目录按 manifest 发现工具。

### 6-7 周（6-27 至 7-06）— 打磨

- 核心路径的 `pytest` 测试。
- 类型注解 + `mypy` 干净通过。

## 风险登记

| 风险 | 缓解 |
| --- | --- |
| `pku3b` 子进程在边界情形下不稳 | 第 1 周即做集成测试；准备桩数据让 UI 任务并行推进 |
| BGE embedder 首次加载耗时 / 内存占用 | 启动期预热；启动闪屏；必要时换量化版本 |
| PyQt6 跨平台打包差异 | 锁定演示机 OS；开发机与演示机同系列 |
| LLM API 配额 / 服务中断（DeepSeek / Kimi） | 缓存高频响应；Provider 可配置切换 |
| 队员可用时间波动（考试、其他课业） | Captain 吸收整合工作；削减顺序作为预约共识 |
| Captain 评审带宽饱和 | 强制小 PR；队员先用 Claude 自检后再提交 |

## 前置假设

- "6 项核心功能在 25 天内均做到实质深度"是 captain 选定的雄心目标，凭借 Claude 辅助实现。按经验估计达成概率约 60%；若到第 2 周中段仍滞后，回退至"宽度优先 + 部分深度"，按上述削减顺序执行。
- 团队成员（`RizzoHou`、`by-lastime`、`Q-star17`）按"协作约定"自由认领任务；不预设固定的功能分块归属。
